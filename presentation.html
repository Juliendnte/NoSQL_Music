01 — Le sujet

Un problème de graphe, résolu avec un outil de graphe

Objectif

  • Rechercher un artiste via MusicBrainz
  • Importer artiste, morceaux, releases, collaborations dans Neo4j — sans doublon
  • Explorer le réseau de collaborations visuellement
  • Produire des statistiques d'analyse du graphe

Pourquoi Neo4j

  • "Qui a collaboré avec qui" = chemins et connexions, pas des tables
  • Cypher exprime nativement ce que le SQL ferait en jointures récursives
  • Comptage de connexions, détection de hubs : direct en Cypher

02 — Modèle de données

6 types de nœuds, 8 relations typées

Genre ←ASSOCIATED_WITH_GENRE Artist FROM_AREA→ Area
COLLABORATED_WITH ⟲ (bidirectionnelle)
Artist PERFORMED / FEATURED_ON→ Recording APPEARS_ON→ Release RELEASED_BY→ Label

Détection des collaborations via les artist-credit MusicBrainz — plus fiable qu'un pattern-matching sur les titres (feat., ft....). Contrainte d'unicité sur mbid pour chaque nœud : zéro doublon, même en réimportant.

06 — Défis techniques

Deux problèmes réels, diagnostiqués et corrigés

Bug corrigé

400 Bad Request systématique sur chaque import.

inc=releases envoyé à l'endpoint browse /recording, où ce paramètre n'est valide qu'en lookup direct. MusicBrainz rejetait 100% des imports. Paramètre inutilisé dans notre code — retiré.

Résilience

MusicBrainz limite à ~1 req/s et pénalise les rafales.

Throttle applicatif partagé + User-Agent identifié. Retry borné côté backend (3 tentatives, ~18s max) et bouton "Réessayer" côté frontend plutôt qu'un chargement infini.
